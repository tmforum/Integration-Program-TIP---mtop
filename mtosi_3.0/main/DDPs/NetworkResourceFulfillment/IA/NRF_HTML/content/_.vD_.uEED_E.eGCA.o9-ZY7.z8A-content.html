Class Call

This object class represents a Call.

The call structure is the abstraction of a call. G.8080 defines a call as the association between endpoints that supports an instance of a service. As such, an end-to-end call is considered to consist of multiple call segments, depending on whether the call traverses multiple domains (signalling, routing or recovery domains included).
For purposes of this interface call and call segments are identical.

The association between calls in a client/server layer relationship are coordinated by the layer NCC (Network Call Controller). This association can recur to mirror a set of "stacked" adaptations.
The target OS interaction with the call is realized through the layer NCC at which the call was requested.
Call Operations can only be done by the target OS that controls the call, this is by the OS that controls the originating points of the call.

A Call can exist in the context of a MultiLayer Subnetwork (MLSN) if the routing domain is equivalent to the MLSN domain.
A Call can exist in the context of a Multi Layer Routing Area (MLRA) if the routing domain is spread over more than one MLSN domain.
In both cases the Call is named by the EMS. It represents the association between endpoints in support of an instance of a service or an infrastructure trail.

A Call is a grouping object that is used to facilitate and manage (establish, modify, delete) a set of Subnetwork Connections (SNCs) that provide end-to-end transport of user information.
Because the Call manages the set of Connections/SNCs as a group, it is possible to take diversity constraints between the individual Connections/SNCs into account when they are created/modified.
Calls which exist in the context of an MLSN manage the SNCs directly.

Calls which exist in the context of an MLRA:
- manage the SNCs indirectly via top-level Connections
- manage the Connections indirectly via top-level Connections
(refer to the Figure entitled "Example of Encapsulation Layer Link segments with off-network CPTPs" in the attached supporting document SD1-44_ConnectionlessTechnologyManagement.pdf).

There is a relationship between Calls and top-level Connections. Zero or more top-level Connections may be associated to a Call. A top-level Connection may not be part of more than one Call.
The bandwidth of a Call is provided by one or more Connections.

Note: Creating a Call between Termination Points (TPs) in the same Managed Element is possible.
Calls are expected to be identified globally.

Traceability: R_TMF518_NRF_I_0085, R_TMF518_NRF_I_0086.

Attributes
String  callId

This attribute shall represent the unique identifier of the Call. The identifier is assigned by the Control Plane and is only used for Control Plane Signalling.
Note: callId corresponds to the Call Name attribute in ITU-T Recommendation G.7713.
CallState  callState

This attribute represents the state of the Call. The Call state dependents on the state of the associated Connections.

Traceability: R_TMF518_NRF_I_0087.
CallEndType  aEnd

This attribute represents the source end point of the Call. The value of this attribute has the following fields:
- SNPP (Control Plane name space)
- SNP (Control Plane name space)
- CPTP/FTP/CTP/PTP (Transport Plane name space)
- TNA Name (which identifies either a TNA or a Group-TNA Name in the Control Plane name space).
All the fields may be empty.

Notes:
All TPs that are managed by different target OSes are off-network TPs which are addressed by remote addresses.
In a Control Plane environment the A-end TP and/or Z-end TP attributes may also contain a "Null-name".
In a Control Plane environment, the type is independent of the type of the Z-end.

For a detailed description of the use of this attribute, see SD1-25 Naming.
CallEndType  zEnd

This attribute represents the destination end point of the Call. The value of this attribute has the following fields:
- SNPP (Control Plane name space)
- SNP (Control Plane name space)
- CPTP/FTP/CTP/PTP (Transport Plane name space)
- TNA Name (which identifies either a TNA or a Group-TNA Name in the Control Plane name space).
All the fields may be empty.

Notes:
All TPs that are managed by different target OSes are off-network TPs which are addressed by remote addresses.
In a Control Plane environment the A-end TP and/or Z-end TP attributes may also contain a "Null-name".
In a Control Plane environment, the type is independent of the type of the A-end.

For a detailed description of the use of this attribute, see SD1-25 Naming.
CallParameterProfile  callParameters

This attribute represents the following parameters:
- Severely Degraded Threshold
- Degraded Threshold
- Class Of Service
- Class Of Service specific Parameters.
DiversityConstraints  callDiversity

This attribute represents the diversity status of the Call.
It provides the information on corouting and diversity (link and node) for the call.
A call may have No Diversity or Link Diversity and/or Node Diversity.
DiversityViolations  diversityViolations

This attribute represents information on the current state of diversity violations of the Call in the following parameters:
- diversitySynthesis
- linkDiversityViolationsList
- linkPartialDiversityList
- nodeDiversityViolationsList
- nodePartialDiversityList.
