 createFloatingTerminationPoint

This operation creates a Floating Termination Point (FTP) within an Equipment (and the CTPs contained in the FTP, if applicable).

Traceability: R_TMF518_RP_II_0090, R_TMF518_RP_II_0091, UC_TMF518_RP_0040.
 createGroupTerminationPoint

This operation creates a new Group Termination Point (GTP). The requesting OS can either specify the list of CTPs comprising the GTP (the list is placed in the tpRefList parameter) or, in the case of contiguous CTPs of the same layerRate, the requesting OS may list the first CTP (in the initialCtpRef parameter) and the number of following CTPs (in the numberOfCtps parameter). The requesting OS may also provide the target OS with the flexibility to decide the list of CTPs by using the gtpEffort parameter. In this flexible mode the target OS uses the tpRefList parameter or the initialCtpRef/ numberOfCtps parameters to determine the total requested bandwidth but it may return a GTP (createdGtp parameter) with a different set of CTPs than those indicated in the original request. Note that this mode allows for the GTP components to be instantiated at a later time by the Managed Element (e.g., upon detection of user's signal). Therefore the operation may successfully return a createdGtp with an empty tpRefList attribute (to be updated at a later time once the component CTPs are created in the Managed Element).

Traceability: R_TMF518_RP_II_0077, R_TMF518_RP_II_0078.

 deleteGroupTerminationPoint

This operation deletes a Group Termination Point (GTP). An attempt to delete a GTP that is involved in a Cross-Connection or SNC should be rejected by the target OS.
The operation is idempotent; i.e., if the operation is called with the name of a non-existent GTP, it will succeed.

Traceability: R_TMF518_RP_II_0079.
 deleteTerminationPointPool

This operation deletes a Termination Point Pool.
The operation is idempotent; i.e., if the operation is called with the name of a non-existent Termination Point Pool, it will succeed.

Traceability: R_TMF518_RP_II_0083.
 modifyGroupTerminationPoint

This operation is used to add or remove CTPs to/from a Group Termination Point (GTP). For a given request, the requesting OS can only add or remove CTPs, not both. It should not be possible to add a CTP that is already involved in a Cross-Connection or SNC, or that is part of another GTP. See attached supporting document SD1-3_BundledSNC for further information on GTP usage.
Attempts to modify a GTP that is involved in a Cross-Connection or SNC should be rejected by the target OS.
The operation is best-effort, i.e., the target OS will add or remove as many of the identified CTPs as possible.
If the operation is called with the name of a non-existent GTP or CTP, it will fail.
Note: If the requesting OS created a GTP with gtpEffort equal to SAME, this also implies that the target OS will not modify on its own, i.e., the target OS will only modify the GTP if requested via the modify GTP operation.

Traceability: R_TMF518_RP_II_0080.
 modifyTerminationPointPool

This operation modifies a Termination Point Pool, i.e., adds/removes TPs or GTPs to/from a TPPool.
For a given request, the requesting OS can only add TPs/GTPs or remove TPs/GTPs, not both. When TPs/GTPs are to be added, all the specified TPs/GTPs shall be contained in Managed Elements of the TP Pool's Subnetwork. It is not possible to add a TP/GTP that is part of another TP Pool or GTP.
The operation is best-effort, i.e., the target OS will add or remove as many of the identified TPs as possible. If the service is called with the name of a non-existent TP Pool or TP or GTP, it will fail.

Traceability: R_TMF518_RP_II_0084.
 setTerminationPointData

This operation can be used to
- terminate and map Connection Termination Points (CTPs); refer to UC_TMF518_RP_0037,
- unterminate and unmap Connection Termination Points (CTPs);refer to UC_TMF518_RP_0038,
- provision Transmission Parameters on a Termination Point; refer to UC_TMF518_RP_0039.
The operation is best effort (except where specified otherwise for a particular parameter). The results of the operation are returned so that the requesting OS is aware of what modifications succeeded.
If the source TP of a broadcast system is used as input, then the entire multipoint system will be affected based on the new parameter values for the source TP.
The tpMappingMode may be set with this operation. When the mode is set to TERMINATED_AND_AVAILABLE_FOR_MAPPING, the target OS will terminate the specified CTP. In this case the target OS must create the specified CTP on the Managed Element if it does not actually exist. Setting the tpMappingMode of an ATM VP or VC CTP can only be done if the CTP has been created. Note that the tpMappingMode can be set only on the ingress and egress CTPs of an ATM SNC since they are the only ones which may not be cross-connected.
The provided Transmission Parameter List is a "delta" list that needs to be applied to the specified TP, i.e. only a subset of the parameters may be specified in the list, in which case only those should be applied in the Managed Element. Transmission Parameters are used to associate a TCA profile with a TP using this service. In this case there are potentially additional failure modes (see Use Case). The assignment of a Transmission Descriptor (TMD) to a Termination Point (TP) as egress or ingress TMD by using the TMD's name amounts to an overwriting of the Layered Transmission Parameters of the TP by the Layered Transmission Parameters of the TMD, and to an overwriting of the additionalInfo parameters of the TP by the additionalObjectInformation parameters of the TMD.
Traceability: R_TMF518_RP_II_0085, R_TMF518_RP_II_0086, R_TMF518_RP_II_0088, R_TMF518_RP_II_0089, R_TMF518_RP_II_0059, UC_TMF518_RP_0030, UC_TMF518_RP_0037, UC_TMF518_RP_0038, UC_TMF518_RP_0039.
