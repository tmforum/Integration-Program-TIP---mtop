Interface CallControlService

Operations
 establishCall

This operation establishes a point to point Call with zero or more top level Connections across one or more Multi Layer Routing Areas (MLRAs). The operation is atomic; i.e., the Control Plane will either establish the entire Call that matches the parameters requested (including all requested Connections) or will fail the entire Call.

The operation can only be used in a top level MLRA.

At least one end point of the Call has to be within the target OS domain. The other end points may be \"off-network\" TPs (i.e., addressed by a remote address) or an SNP/SNPP/TNA not directly managed by the local target OS.

If routing constraints are specified for the associated Connections, they shall match the constraints based to the degree mandated by the appropriate effort statement.

For more detail on the process of establishing a call in the Control Plane, please see SD1-45_ASONControlPlaneManagement-Primer and SD1-46_ASONControlPlaneManagement-Scenarios.

Traceability: R_TMF518_RP_II_0108, R_TMF518_RP_II_0109, UC_TMF518_RP_0067.
 modifyCall

The operation is used to modify only the attributes of the Call. The bandwidth is modified through other operations. The operation is best effort except the change of the user label with forced user label uniqueness.

This operation is able to modify the following attributes of a Call:
- userLabel
- owner
- networkAccessDomain
- additionalInfo.

Traceability: R_TMF518_RP_II_0111, R_TMF518_RP_II_0112, UC_TMF518_RP_0071.
 releaseCall

This operation releases a point-to-point Call with zero or more top level Connections across one or more Multi Layer Routing Areas (MLRAs). The operation is directed against the top level MLRA which contains the Call. As a result of this operation the Call and the supporting (top level) Connections and their Subnetwork Connections will be deleted from the network. If the Call has an associated Topological Link in the client technology, this Link will also be deleted.
If the target OS is able to determine that the Call is still carrying traffic, the operation is rejected.

Traceability: R_TMF518_RP_II_0110, UC_TMF518_RP_0068.
 addConnections

This operation adds one or more Connections or Subnetwork Connections to an existing Call. If necessary, it is also possible to set any TP transmission parameters.
In a Control Plane environment it is assumed that the Control Plane will either set up the entire top level Connection or will fail the entire top level Connection; there is no partial top level Connection state.
If the requesting OS specifies routing constraints for the Connections or Subnetwork Connections in the request they should match the constraints based to the degree mandated by the appropriate effort statement. If the target OS cannot satisfy the routing criteria provided by the requesting OS, within the boundaries of the effort level requested, an exception is raised.

Traceability: R_TMF518_RP_II_0113, R_TMF518_RP_II_0114, UC_TMF518_RP_0069.
 removeConnections

This operation removes one or more supporting Connections (i.e., top level Connections or Subnetwork Connections) from an existing Call. If necessary, it is also possible to set any TP transmission parameters.
In the non Control Plane case the target OS has to make the effort to remove any partial SNCs resulting from this operation. If this is not possible, the target OS will return the list of SNCs which could not be deleted.

Traceability: R_TMF518_RP_II_0115, UC_TMF518_RP_0070.
 modifyDiversityAndCorouting

This operation sets/modifies (including nulling) diversity and co-routing parameters on an existing Call and it's supporting Connections.
In a Control Plane environment, if the target OS accepts the operation where rerouting is required, one or more Connections may enter the "Searching" state. There will be no rollback and the Connections may remain in the "Searching" state.
In a non Control Plane environment, if the target OS accepts the operation where rerouting is required, one or more SNCs may enter the "Partial" state.
If all diversity requirement parameters ("Link Diversity level of Effort", "Node Diversity level of effort", "Co-routing level of effort") are set to "None" and "Number of Route Groups" is "NA" or "0", the target OS will remove all diversity and co-routing requirements from the Call and the Route Group labels of the Connections will be set to an empty value.

Note:
If any of the following is true:
- "Co-routing level of effort" is not "None"
- "Link Diversity level of effort" is not "None"
- "Node Diversity level of effort" is not "None" and
- "Number of Route Groups" is "NA"
then each Connection of the Call must have a "Route Group Label" specified (i.e. of non-Null value).

Traceability: R_TMF518_RP_II_0116, R_TMF518_RP_II_0117, R_TMF518_RP_II_0118, R_TMF518_RP_II_0119, UC_TMF518_RP_0072.
 addRoute??

This operation creates a new route (e.g. for restoration purposes) and associates it to the SNC. The route is created in LOCKED state.
A route belongs to only one SNC. However CCs/CTPs can be shared by routes of different SNCs.
A route applies to CCs at the same layer of the SNC. The route identifier must be a unique identifier within the SNC name, with format up to target OS.
It is possible to specify if the creating route is the intended route or not; if intended, then the former intended route is updated to backup route. Only one intended route can be associated to a SNC at a time.
It is possible to specify if the route to be created is exclusive or not; if exclusive, then the target OS must find a route that does not conflict or share CCs with any other existing SNC route, in any administrative state. Once an exclusive route has been created by the target OS, any further creation operation which conflicts with the exclusive route shall be rejected.
The route implicitly inherits from its SNC the following attributes:
- the endpoints
- owner
- direction
- rerouteAllowed
- networkRouted
- connectionRate
- sncType
- staticProtectionLevel
- protectionEffort is considered EFFORT_SAME, i.e. if target OS can not create a route with same static protection level as the SNC, the operation is rejected.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

Additional information on exceptions:
protectionEffortNotMet - Raised when the requesting OS requests a route with a static protection level (inherited from SNC) that cannot be met by the target OS.
unableToComply - Raised when the target OS is unable to find an appropriate route.
Raised when the target OS can not meet the tolerableImpact requested by the requesting OS.
Raised when the target OS is unable to execute the request because at least one of the parameters although valid can not be set and that parameter is identified as "not best effort" in additionalInfo.
objectInUse - Raised when the route is in conflict with an \"exclusive\" (even locked) route of another SNC.

Traceability: R_TMF518_RP_II_xxx, UC_TMF518_RP_xxx.
 removeRoute??

The operation allows an NMS to request the deletion of a route of the subnetwork connection.
It is possible to delete a locked backup route which is \"inUseBy\" other routes of other SNCs, because this operation has no side effect on routes of any other SNCs, even if sharing CCs/TPs.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

In case of failure:
The route will not be deleted on the EMS. The errorReason field of the exception will contain the reason for the failure.

Operation Exceptions:
1. NOT_IMPLEMENTED - Raised when an EMS does not support this operation.
2. INTERNAL_ERROR - Raised in case of non-specific EMS internal failure.
3. INVALID_INPUT - Raised when any input parameter is syntactically incorrect.
4. ENTITY_NOT_FOUND - Raised when routeId reference an object that does not exist.
5. NOT_IN_VALID_STATE - Raised when the route is in the UNLOCKED state, or if the route is the intended one.
6. UNABLE_TO_COMPLY - Raised when the EMS is unable to execute the request because at least one of the
parameters although valid can not be set and that parameter is identified as \"not best effort\" in additionalInfo.
If the EMS cannot determine the reason it could not comply, it is also allowed to throw UNABLE_TO_COMPLY.
7. NE_COMM_LOSS - Raised when communication to managed element is lost and this prevents the deletion of the route.
 setIntendedRoute??

The operation allows an NMS to request to set the addressed route as intended route. The formerly intended route (if different from addressed) is no longer the intended one, in order to respect the rule that an SNC must have one intended route, and only one. The addressed route can be in any administrative / actual state.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

In case of failure:
The errorReason field of the exception will contain the reason for the failure.

Operation Exceptions:

 setRoutesAdminState??

The operation allows an NMS to specify a list of routes jointly with their provisioned administrative state. If all routes of the SNC are successfully locked, then the SNC state transits to PENDING. If the SNC state was PARTIAL or PENDING, and at least one route is successfully unlocked, then the SNC state may transits to ACTIVE.
See state change diagram of sncState.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

Operation Exceptions:

2. INTERNAL_ERROR - Raised in case of non-specific EMS internal failure.
3. INVALID_INPUT - Raised when any input parameter is syntactically incorrect.
4. ENTITY_NOT_FOUND - Raised when at least one route identifiers reference an object that does not exist.
5. NE_COMM_LOSS - Raised when communication to managed element is lost and this prevents any change to the SNC.
 switchRoute??

The operation is used to activate the input route and deactivate the current (or partial) route(s), if different from input route.
The operation does not affect the administrativeState of any route. The restoration process may re-route again, e.g. in case of failures.

In case of failure:
No exception thrown, except in the cases listed below. If the SNC or any of its network resources have changed as a result of this operation, then no exception can be thrown so that the sncState can be passed back to the NMS. The resulting sncState will be either PARTIAL or ACTIVE.
The sncState will be PARTIAL if not all cross-connects (of given input route) on MEs have been successfully set up, and there is not an active route. Moreover one or more of the cross-connects to be removed may be still be active, this leads to two (or more) routes in PARTIAL state.
The sncState will be ACTIVE if all cross-connects (of given input route) on MEs have been successfully set up, but one or more of the cross-connects (of any of the other partial routes) to be removed are still active, this leads to one route in ACTIVE state and another (or more) route in PARTIAL state. The errorReason parameter is set accordingly.

Recovery scenarios: this operation can be performed again addressing
- an already active route, in order to retry the deactivation of other route(s) in partial state
- a partial route, in order to retry its activation (and retry, if needed, the deactivation of other partial routes as well).

The EMS will apply the transmission parameters specified in the tpsToModify parameter (same behaviour as setTPData). They may be applied before or after the creation of the CCs, as appropriate. Existing TP transmission parameters for which no changes were requested in tpsToModify will be left unchanged. However, the alarm reporting on the CTPs and the containing TPs may be turned on by the EMS as part of this request, unless otherwise specified via the transmission parameter \"AlarmReporting\". A similar behaviour is also allowed for the serviceState parameter, which may be set to IN_SERVICE for the CTPs and the containing TPs. See TransmissionParameters object.
If a given entry in the list of transmission parameters specified in the TPData_T can not be successfully applied to the TP, for any reason, then the errorReason field is appended with an appropriate reason text. Applying transmission parameters is best-effort and the resulting values of the transmission parameters are provided in the updated tpsToModify parameter.

An already active SNC route can be activated again; the EMS is allowed to not send the commands to the ME a second time for the cross connect establishment however the commands may be sent for the transmission parameters.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

Operation Exceptions:
1. NOT_IMPLEMENTED - Raised when the EMS does not support this operation.
2. INTERNAL_ERROR - Raised in case of non-specific EMS internal failure.
3. INVALID_INPUT - Raised when any input parameter (e.g. tpsToModify) is syntactically incorrect.
4. OBJECT_IN_USE - Raised when the route can not be activated because of CC or CTP conflicts with an
active route of another SNC (with equal or higher priority).
5. ENTITY_NOT_FOUND - Raised when routeId or tpsToModify reference a route/TP object that does not exist.
6. NE_COMM_LOSS - Raised when communication to managed element is lost and this prevents any change to
the SNC.
7. NOT_IN_VALID_STATE - Raised where the SNC would involve a CTP that is not connectable due to the state of
the server TP or in the case of inverse multiplexing where the SNC would involve a CTP that is not connectable
due to the state of the client TP.
8. UNABLE_TO_COMPLY - Raised when the EMS can not meet the tolerableImpact requested by the NMS.
Raised when the EMS is unable to execute the request because at least one of the parameters although valid can not
be set and that parameter is identified as \"not best effort\" in the supporting document SD1-16_LayeredParameters.pdf
associated to the TransmissionParameters object or in the additionalInfo attribute; i.e. the whole operation will be
rejected if one parameter fails that must be set. This is potentially usable for cases where a parameter must be set for
the SNC to be valid.
Raised if the route cannot be activated because the EMS cannot comply for a reason different than the ones above.
If the EMS cannot determine the reason it could not comply, it is also allowed to throw UNABLE_TO_COMPLY.

 releaseCall 
Public  releaseCall(, String, ,  )

This operation releases a point-to-point Call with zero or more top level Connections across one or more Multi Layer Routing Areas (MLRAs). The operation is directed against the top level MLRA which contains the Call. As a result of this operation the Call and the supporting (top level) Connections and their Subnetwork Connections will be deleted from the network. If the Call has an associated Topological Link in the client technology, this Link will also be deleted.
If the target OS is able to determine that the Call is still carrying traffic, the operation is rejected.

Traceability: R_TMF518_RP_II_0110, UC_TMF518_RP_0068.

Parameters:
callRef [1]This parameter provides the name of the Call to be released.

The operation allows an NMS to request the deletion of a route of the subnetwork connection.
It is possible to delete a locked backup route which is \"inUseBy\" other routes of other SNCs, because this operation has no side effect on routes of any other SNCs, even if sharing CCs/TPs.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

In case of failure:
The route will not be deleted on the EMS. The errorReason field of the exception will contain the reason for the failure.

 switchRoute?? 
Public  switchRoute??(, , , , , ,  )

The operation is used to activate the input route and deactivate the current (or partial) route(s), if different from input route.
The operation does not affect the administrativeState of any route. The restoration process may re-route again, e.g. in case of failures.

In case of failure:
No exception thrown, except in the cases listed below. If the SNC or any of its network resources have changed as a result of this operation, then no exception can be thrown so that the sncState can be passed back to the NMS. The resulting sncState will be either PARTIAL or ACTIVE.
The sncState will be PARTIAL if not all cross-connects (of given input route) on MEs have been successfully set up, and there is not an active route. Moreover one or more of the cross-connects to be removed may be still be active, this leads to two (or more) routes in PARTIAL state.
The sncState will be ACTIVE if all cross-connects (of given input route) on MEs have been successfully set up, but one or more of the cross-connects (of any of the other partial routes) to be removed are still active, this leads to one route in ACTIVE state and another (or more) route in PARTIAL state. The errorReason parameter is set accordingly.

Recovery scenarios: this operation can be performed again addressing
- an already active route, in order to retry the deactivation of other route(s) in partial state
- a partial route, in order to retry its activation (and retry, if needed, the deactivation of other partial routes as well).

The EMS will apply the transmission parameters specified in the tpsToModify parameter (same behaviour as setTPData). They may be applied before or after the creation of the CCs, as appropriate. Existing TP transmission parameters for which no changes were requested in tpsToModify will be left unchanged. However, the alarm reporting on the CTPs and the containing TPs may be turned on by the EMS as part of this request, unless otherwise specified via the transmission parameter \"AlarmReporting\". A similar behaviour is also allowed for the serviceState parameter, which may be set to IN_SERVICE for the CTPs and the containing TPs. See TransmissionParameters object.
If a given entry in the list of transmission parameters specified in the TPData_T can not be successfully applied to the TP, for any reason, then the errorReason field is appended with an appropriate reason text. Applying transmission parameters is best-effort and the resulting values of the transmission parameters are provided in the updated tpsToModify parameter.

An already active SNC route can be activated again; the EMS is allowed to not send the commands to the ME a second time for the cross connect establishment however the commands may be sent for the transmission parameters.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

Operation Exceptions:
1. NOT_IMPLEMENTED - Raised when the EMS does not support this operation.
2. INTERNAL_ERROR - Raised in case of non-specific EMS internal failure.
3. INVALID_INPUT - Raised when any input parameter (e.g. tpsToModify) is syntactically incorrect.
4. OBJECT_IN_USE - Raised when the route can not be activated because of CC or CTP conflicts with an
active route of another SNC (with equal or higher priority).
5. ENTITY_NOT_FOUND - Raised when routeId or tpsToModify reference a route/TP object that does not exist.
6. NE_COMM_LOSS - Raised when communication to managed element is lost and this prevents any change to
the SNC.
7. NOT_IN_VALID_STATE - Raised where the SNC would involve a CTP that is not connectable due to the state of
the server TP or in the case of inverse multiplexing where the SNC would involve a CTP that is not connectable
due to the state of the client TP.
8. UNABLE_TO_COMPLY - Raised when the EMS can not meet the tolerableImpact requested by the NMS.
Raised when the EMS is unable to execute the request because at least one of the parameters although valid can not
be set and that parameter is identified as \"not best effort\" in the supporting document SD1-16_LayeredParameters.pdf
associated to the TransmissionParameters object or in the additionalInfo attribute; i.e. the whole operation will be
rejected if one parameter fails that must be set. This is potentially usable for cases where a parameter must be set for
the SNC to be valid.
Raised if the route cannot be activated because the EMS cannot comply for a reason different than the ones above.
If the EMS cannot determine the reason it could not comply, it is also allowed to throw UNABLE_TO_COMPLY.

Parameters:
additionalInfo [1]/* inout */ Allows the communication of additional information which is not explicitly modelled.
emsFreedomLevel [1]/* in */ The maximum level of freedom allowed to the EMS to perform the operation.
errorReason [1]/* out */ Specifies the activation error(s) if any.
routeId [1]/* in */ The identifier of the route.
sncState [1]/* out */ The state of the subnetwork connection after the operation.
tolerableImpact [1]/* in */ This parameter indicates the maximum tolerable impact allowed.
tpsToModify [1]/* inout */ A list of TPs and parameters to apply, updated to provide the resulting parameters.
Returns:
<Unnamed Parameter> [1]

<Unnamed Constraint> = "1. sncState is set to ACTIVE. 2. All cross-connects required for the given route are activated successfully. 3. If needed, all the cross-connects of former current (or partial) route(s) are deactivated successfully."
