Interface ConnectionControlService

Operations
 activateSubnetworkConnection

This operation is used to put an already existing Subnetwork Connection (SNC) into the ACTIVE state. It can also be applied to bundled SNCs.
More precisely, all the SNC routes, intended and backup, if any, are unlocked. The restoration process will take control of the unlocked routes and activate the proper one.
Note: The behaviour of SNC State is not dependent on the number of routes associated to the SNC.

Traceability: R_TMF518_RP_II_0007, R_TMF518_RP_II_0008, UC_TMF518_RP_0002.
 addRoute

This operation creates a new Route (e.g. for restoration purposes) and associates it to the given SNC. The route is created in LOCKED state.
A route belongs to only one SNC. However CCs/CTPs can be shared by Routes of different SNCs.
A route applies to CCs at the same layer of the SNC. The route identifier must be an unique identifier within the SNC name, with format up to target OS.
It is possible to specify if the Route to be created is the intended Route or not; if intended, then the former intended Route is updated to backup Route. Only one intended Route can be associated to a SNC at a time.
It is possible to specify if the Route to be created is EXCLUSIVE or not; if EXCLUSIVE, then the target OS must find a Route that does not conflict or share CCs with any other existing SNC Route, in any administrative state. Once an exclusive Route has been created by the target OS, any further creation operation which conflicts with the exclusive Route shall be rejected.
The Route implicitly inherits the following attributes from its SNC:
- the endpoints
- owner
- direction
- rerouteAllowed
- networkRouted
- connectionRate
- type
- staticProtectionLevel
- protectionEffort is considered EFFORT_SAME, i.e. if target OS can not create a Route with same static protection level as the SNC, the operation is rejected.
Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study and is not applicable for v3.0.

Traceability: R_TMF518_RP_II_0011, R_TMF518_RP_II_0012, UC_TMF518_RP_0004.
 checkValidSubnetworkConnection

This operation checks whether a valid Subnetwork Connection (SNC) can be created based on input parameters without actually creating it.
The test should check for the existence of hardware that will support the requested SNC. If the mustConsiderResources parameter is false, the check must be independent of the current specific resource usage in the Multi Layer Subnetwork (as in createSubnetworkConnection). If the mustConsiderResources parameter is true, the check must consider the current specific resource usage in the Multi Layer Subnetwork (as in activateSubnetworkConnection); in that case, the rules of the target OS' mode of operation apply to the check (see supporting document SD1-23_ModesOfOperation).
This operation also applies to bundled SNCs.

Traceability: R_TMF518_RP_II_0006.
 createAndActivateSubnetworkConnection

This operation creates and activates a Subnetwork Connection. The operation also applies to bundled SNCs (see supporting document SD1-3_BundledSNC for further details).
It conceptually behaves like a call to createSubnetworkConnection followed by a call to activateSubnetworkConnection.

Traceability: R_TMF518_RP_II_0009, R_TMF518_RP_II_0010, UC_TMF518_RP_0003, UC_TMF518_RP_0047.
 createModifiedSubnetworkConnection

This operation modifies the Route of the addressed Subnetwork Connection (SNC).
If the target OS does preserve the SNC name, the addressed SNC Route is modified. If the SNC was in PENDING or PARTIAL state, then the state is unchanged. If the SNC was in ACTIVE state, then the output state is PARTIAL. In case the SNC has several routes, then the administrative state of the addressed route will always transit to LOCKED state.
If the target OS does not preserve the SNC name, then the operation creates a new pending SNC from an existing pending or active SNC. It is similar to createSubnetworkConnection except that the created SNC is made based on an existing one.
This operation also applies to bundled SNCs.

Traceability: R_TMF518_RP_II_0015, R_TMF518_RP_II_0016, UC_TMF518_RP_0006.
 createSubnetworkConnection

This operation creates a new Subnetwork Connection (SNC).
The SNC is created with its intended Route, in LOCKED state. By means of other operations it is possible to add (then remove) additional Routes for restoration purposes.
The operation also applies to bundled SNCs. See supporting document SD1-3_BundledSNC for further details.
At a CM end point the span of the SNC starts with a fixed or flexible connection through the Managed Element at the SNC layer, i.e. the connectable layer of the end point. At an LC end point the span of the SNC starts with a G.805 link connection, i.e. connectivity which is adapted from (i.e., provided by) a server layer, e.g. from/by a Topological Link.

Traceability: R_TMF518_RP_II_0002, R_TMF518_RP_II_0003, UC_TMF518_RP_0001.
 deactivateAndDeleteSubnetworkConnection

This operation deactivates and deletes a Subnetwork Connection.
It conceptually behaves like a call to deactivateSubnetworkConnection followed by a call to deleteSubnetworkConnection.
This operation also applies to bundled SNCs.

Traceability: R_TMF518_RP_II_0030, R_TMF518_RP_II_0031, R_TMF518_RP_II_0032, UC_TMF518_RP_0009.
 deactivateSubnetworkConnection

This operation deactivates a Subnetwork Connection.
If the SNC has more alternative Routes, then the operation locks all the SNC routes, intended and backup.
This operation also applies to bundled SNCs.

Traceability: R_TMF518_RP_II_0026, R_TMF518_RP_II_0027, UC_TMF518_RP_0007.
 deleteSubnetworkConnection

This operation deletes a Subnetwork Connection (SNC). If the SNC has more alternative Routes, then the operation deletes the SNC, its intended and all backup Route(s).
This operation also applies to bundled SNCs. In the case that the mustRemoveGtps attribute is set to "true" and the target OS cannot remove all the interior GTPs, the target OS should return a list of the GTPs that could not be removed in the errorReason output parameter. The errorReason will be "not all interior GTPs could be deleted" - this will be followed by the names of the GTPs that could not be deleted.

Traceability: R_TMF518_RP_II_0028, R_TMF518_RP_II_0028, UC_TMF518_RP_0008.
 modifySubnetworkConnection

This operation modifies and activates a Subnetwork Connection (SNC) on the network. If the target OS does not preserve the SNC name, then newSNC will have a different name than formerly addressed SNC, which is deactivated and deleted.
Behavior is as createModifiedSubnetworkConnection to find the Route and swapSubnetworkConnection to set it on the network.
This operation also applies to bundled SNCs.

Traceability: R_TMF518_RP_II_0017, R_TMF518_RP_II_0016.
 removeRoute

This operation deletes a Route from a Subnetwork Connection.
It is possible to delete a locked backup Route which is "inUseBy" other Routes of other SNCs, because this operation has no side effect on Routes of any other SNCs, even if sharing Cross-Connects/TPs.
Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study.

Traceability: R_TMF518_RP_II_0013, R_TMF518_RP_II_0014, UC_TMF518_RP_0005.
 setIntendedRoute

This operation sets the addressed Route as intended Route. The formerly intended Route (if different from addressed) is no longer the intended one, in order to respect the rule that an SNC must have one intended Route, and only one. The addressed Route can be in any administrative / actual state.
Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study.

Traceability: R_TMF518_RP_II_0024, R_TMF518_RP_II_0025.
 setRoutesAdminState

This operation sets the administrative state of a list of Routes of a Subnetwork Connection (SNC). If all Routes of the SNC are successfully locked, then the SNC state transits to PENDING. If the SNC state was PARTIAL or PENDING, and at least one Route is successfully unlocked, then the SNC state may transits to ACTIVE.
See state change diagrams of sncState in NRF:SubNetworkConnection.

Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study.

Traceability: R_TMF518_RP_II_0022, R_TMF518_RP_II_0023.
 swapSubnetworkConnection

This operation activates a pending Subnetwork Connection (SNC) and deactivates an active SNC.
It also applies to bundled SNCs.

Traceability: R_TMF518_RP_II_0018, R_TMF518_RP_II_0019.
 switchRoute

This operation is used to activate the Route provided and deactivate the current (or partial) Route(s), if different from the provided Route.
The operation does not affect the administrative state of any Route. The restoration process may re-route again, e.g. in case of failures.
In case of failure:
No exception thrown, except in the cases listed below. If the SNC or any of its network resources have changed as a result of this operation, then no exception can be thrown so that the SNC state can be passed back to the requesting OS. The resulting SNC state will be either PARTIAL or ACTIVE.
The SNC state will be PARTIAL if not all Cross-Connects (of the provided Route) on the Managed Elements have been successfully set up, and there is not an active Route. Moreover one or more of the Cross-Connects to be removed may be still be active, this leads to two (or more) Routes in PARTIAL state.
The SNC state will be ACTIVE if all Cross-Connects (of the provided Route) on the Managed Elements have been successfully set up, but one or more of the Cross-Connects (of any of the other partial Routes) to be removed are still active, this leads to one Route in ACTIVE state and another (or more) Route in PARTIAL state. The errorReason parameter is set accordingly.
Recovery scenarios: This operation can be performed again addressing
- an already active Route, in order to retry the deactivation of other Route(s) in partial state
- a partial Route, in order to retry its activation (and retry, if needed, the deactivation of other partial Routes as well).
The target OS will apply the Transmission Parameters specified in the tpDataListToModify parameter (same behaviour as setTerminationPointData). They may be applied before or after the creation of the CCs, as appropriate. Existing TP Transmission Parameters for which no changes were requested in tpDataListToModify will be left unchanged. However, the alarm reporting on the CTPs and the containing TPs may be turned on by the target OS as part of this request, unless otherwise specified via the Transmission Parameter "AlarmReporting". A similar behaviour is also allowed for the serviceState parameter, which may be set to IN_SERVICE for the CTPs and the containing TPs. See TransmissionParameters object.
If a given entry in the list of Transmission Parameters specified in the TerminationPointData can not be successfully applied to the TP, for any reason, then the errorReason field is appended with an appropriate reason text. Applying Transmission Parameters is best-effort and the resulting values of the Transmission Parameters are provided in the updated tpDataListToModify parameter.
An already active SNC Route can be activated again; the target OS is allowed to not send the commands to the Managed Element a second time for the Cross-Connect establishment however the commands may be sent for the Transmission Parameters.
Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study.

Traceability: R_TMF518_RP_II_0020, R_TMF518_RP_II_0021.

 switchRoute 
Public  switchRoute(, String, FreedomLevel, String, , , GradesOfImpact,  )

This operation is used to activate the Route provided and deactivate the current (or partial) Route(s), if different from the provided Route.
The operation does not affect the administrative state of any Route. The restoration process may re-route again, e.g. in case of failures.
In case of failure:
No exception thrown, except in the cases listed below. If the SNC or any of its network resources have changed as a result of this operation, then no exception can be thrown so that the SNC state can be passed back to the requesting OS. The resulting SNC state will be either PARTIAL or ACTIVE.
The SNC state will be PARTIAL if not all Cross-Connects (of the provided Route) on the Managed Elements have been successfully set up, and there is not an active Route. Moreover one or more of the Cross-Connects to be removed may be still be active, this leads to two (or more) Routes in PARTIAL state.
The SNC state will be ACTIVE if all Cross-Connects (of the provided Route) on the Managed Elements have been successfully set up, but one or more of the Cross-Connects (of any of the other partial Routes) to be removed are still active, this leads to one Route in ACTIVE state and another (or more) Route in PARTIAL state. The errorReason parameter is set accordingly.
Recovery scenarios: This operation can be performed again addressing
- an already active Route, in order to retry the deactivation of other Route(s) in partial state
- a partial Route, in order to retry its activation (and retry, if needed, the deactivation of other partial Routes as well).
The target OS will apply the Transmission Parameters specified in the tpDataListToModify parameter (same behaviour as setTerminationPointData). They may be applied before or after the creation of the CCs, as appropriate. Existing TP Transmission Parameters for which no changes were requested in tpDataListToModify will be left unchanged. However, the alarm reporting on the CTPs and the containing TPs may be turned on by the target OS as part of this request, unless otherwise specified via the Transmission Parameter "AlarmReporting". A similar behaviour is also allowed for the serviceState parameter, which may be set to IN_SERVICE for the CTPs and the containing TPs. See TransmissionParameters object.
If a given entry in the list of Transmission Parameters specified in the TerminationPointData can not be successfully applied to the TP, for any reason, then the errorReason field is appended with an appropriate reason text. Applying Transmission Parameters is best-effort and the resulting values of the Transmission Parameters are provided in the updated tpDataListToModify parameter.
An already active SNC Route can be activated again; the target OS is allowed to not send the commands to the Managed Element a second time for the Cross-Connect establishment however the commands may be sent for the Transmission Parameters.
Note: The relationship between Bundled SNC service and the SNC with Multiple Routes feature requires further study.

Traceability: R_TMF518_RP_II_0020, R_TMF518_RP_II_0021.
