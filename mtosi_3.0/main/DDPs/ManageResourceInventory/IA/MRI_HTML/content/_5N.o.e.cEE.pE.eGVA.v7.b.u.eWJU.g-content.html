Interface CallConnectionRetrievalService

Operations
 getAllCallsAndTopLevelConnections

This operation provides a way for the requesting OS to retrieve all details of the Calls and associated Top Level Connections in the scope of a specified top level MLRA.
A target OS will only return the Calls and top level Connections that it controls the Originating Point of. By implication this informs the requesting OS which target OS controls each Call.

Note:
The reply can be an empty list, if there are no Calls established from any of the points controlled by the target OS.

Traceability: R_TMF518_MRI_II_0157.
 getAllCallsAndTopLevelConnectionsAndSNCs

This operation provides a way for the requesting OS to retrieve the attributes of all the Calls associated with an MLRA. For each Call object it returns also the attributes of all associated top-level Connection objects. For each top-level Connection object it returns also the attributes of all supporting SubnetworkConnection objects.

Additional information on exceptions:
If the requesting OS provides a callId AND a callRef, the target OS must reject the operation with INVALID_INPUT.

Traceability: R_TMF518_MRI_II_0159.
 getAllCallsAndTopLevelConnectionsWithME

This operation allows the requesting OS to retrieve the attributes of all the Calls associated with an MLRA and supported by an ME. For each Call object it returns also the attributes of all associated top-level Connection objects. For each top-level Connection object it returns also the attributes of all supporting SubnetworkConnection objects.

This operation provides a way for the requesting OS to return all the Calls managed by the target OS within the scope of the top level MLRA (with all the attributes for each Call) and the top level Connections for each Call (with all the attributes for each top level Connection) that are supported by the specified managed element.
This operation has exactly the same behaviour as getAllCallsAndTopLevelConnections(), except that the request returns only Calls (and all Top Level Connections) that have at least one Top Level connection terminated on an SNP that corresponds to a TP contained in the specified ME.

Additional information on exceptions:
invalidInput - Raised if the mlraRef does not represent an MLRA in the Control Plane.
unableToComply - Raised if the specified MLRA is not a Top Level MLRA.

Traceability: R_TMF518_MRI_II_0165.
 getAllCallsAndTopLevelConnectionsAndSNCsWithME

This operation provides a way for the requesting OS to retrieve the attributes of all the Calls associated with an MLRA and supported by an ME. For each Call object it returns also the attributes of all associated top-level Connection objects. For each top-level Connection object it returns also the attributes of all supporting SubnetworkConnection objects.
This operation has exactly the same behaviour as getAllCallsAndTopLevelConnectionsAndSNCs(), except that the request returns only Calls (and all Top Level Connections with their supporting SNCs) that have at least one Top Level connection terminated on the specified ME.

Additional information on exceptions:
unableToComply - Raised if the specified MLRA is not managed by the Control Plane.

Traceability: R_TMF518_MRI_II_0160.
 getAllCallsAndTopLevelConnectionsAndSNCsWithTP

This operation provides a way for the requesting OS to retrieve the attributes of all Calls associated with an MLRA and having at least one Connection/Subnetwork Connection which terminates on the specified TP itself or terminates on any TP contained in the specified TP. For each Call object it returns also the attributes of all associated top-level Connection objects. For each top-level Connection object it returns also the attributes of all supporting SubnetworkConnection objects.
This operation has exactly the same behaviour as getAllCallsAndTopLevelConnectionsAndSNCs(), except that the request returns only Calls (and all Top Level Connections with their supporting SNCs) that have at least one Top Level connection supported by the specified TP (either directly or via a TP contained in this TP).

Traceability: R_TMF518_MRI_II_0161.
 getAllCallIdsWithTP

This operation provides a way for the requesting OS to retrieve the ID of all Calls supported by the specified TP.
A Call is considered as using the specified TP if any of its supporting connections use an SNP that corresponds to the specified TP or a TP contained in the specified TP.

Note that this operation only returns calls ID and not other details as it is only the ID that will be available to the control plane.
In fact this operation applies only in control plane context. For example if a PTP in the centre of the network is referenced, the control plane will be aware of the ID of the calls that pass through the resources of that PTP that have been allocated to the control plane but will not be aware of any other details for calls that originate outside the routing area to which the resources from the PTP are allocated.

Traceability: R_TMF518_MRI_II_0163.
 getAllCallIdsWithSNPPOrTNAName

This operation provides a way for the requesting OS to retrieve the ID of all the Calls that are supported by either the specified SNPP or the specified TNA.
This operation has exactly the same behaviour as getAllCallIdsWithTP, but instead of using TP as an input, it specifies a SNPP or TNA.

Additional information on exceptions:
unableToComply - Raised if the specified input is not controlled by the Control Plane.

Traceability: R_TMF518_MRI_II_0163.
 getCallAndTopLevelConnectionsAndSNCs

This operation provides a way for the requesting OS to retrieve the attributes of a Call given a specific Call Id or a Call name that is within the scope of the top-level MLRA. The operation returns also the attributes of all associated top-level Connection objects. For each top-level Connection object it returns the attributes of all supporting SubnetworkConnection objects.
This operation has exactly the same behaviour as getAllCallsAndTopLevelConnectionsAndSNCs(), except that the request returns only the named call (and all Top Level Connections with their supporting SNCs).

Additional information on exceptions:
If the requesting OS provides a callId AND a callRef, the target OS must reject the operation with INVALID_INPUT.
unableToComply - Raised if the specified Call is defined in an MLRA which is not managed by the Control Plane.

Traceability: R_TMF518_MRI_II_0162.
 getCallAndTopLevelConnections

This operation provides a way for the requesting OS to retrieve the attributes of a Call given a specific Call Id or a Call name that is within the scope of the top-level MLRA. The operation returns also the attributes of all associated top-level Connection objects.
This operation has exactly the same behaviour as getAllCallsAndTopLevelConnections(), except that the request returns only the call which is identified by callId or named by callName (and all Top Level Connections).

Additional information on exceptions:
If the requesting OS provides a callId AND a callRef, the target OS must reject the operation with INVALID_INPUT.
unableToComply - Raised if the specified Call is not defined in a Top Level MLRA, controlled by the Control Plane.

Traceability: R_TMF518_MRI_II_0158.
 getCall

This operation returns the attributes of a Call object given a specified Call name.
Note: The target OS has to control the originating end of the Call otherwise the Call information is not available.

Additional information on exceptions:
entityNotFound - Raised when callRef references a Call object that does not exist (e.g. when the target OS does not control the originating end, the Call information is not available at this target OS).

Traceability: R_TMF518_MRI_II_0164.

 getRoute

This operation returns the route for the Connection. The route information, in the form of cross-connections, should identify the resources allocated to the Connection at the time of the request and the way in which these resources are used.
If the Connection has more alternative routes, then it retrieves the intended route if the Connection is partial, otherwise it retrieves the active route. This because there can be more partial routes and no active route, in such case the intended route is replied, as it is not known which of the partial routes is the candidate for becoming the active route.
The requesting OS specifies if it wants to retrieve only resources in the layer of the Connection (the cross-connections that belong to the Connection), or higher order cross-connections also (the cross-connections of other Connections that are used to carry the traffic of the queried connection).
All cross-connections allocated to the connection must be returned, even if they are not active in the managed elements.

This allows the requesting OS to query the route that will be used before activating an SNC.

Traceability: R_TMF518_MRI_II_0175, UC_TMF518_MRI_0027.
 getConnectionsAndRouteDetails??

This operation returns the routes of the Connections that support top level Connections of the specified Call.

The request may be tailored by the following allowed input combinations:

In the case of a 2 level routing area hierarchy:
- CallID only: the target OS should return the top level connections supporting the Call and their detailed routes.
- CallID and sNPOrSNPPID: The target OS should return the routes for the Connection(s) that originate from the referenced SNP or SNPP for the referenced Call. If the SNP or SNPP is at the hierarchical level of a Routing Node, then the response is an empty list.

In the case of a 3 level routing area hierarchy:
- CallID only: The target OS should return the routes for all connections that support any top level Connection of the specified Call for all MLRAs directly managed by the target OS.
- CallID and mLRAName: The target OS should return the routes for all connections within the referenced MultiLayerRoutingArea that supports any top level Connections of the referenced Call (there could be more than one top level Connection for the same Call passing through a MultiLayerRoutingArea). If the referenced MultiLayerRoutingArea is a Routing Node, then the response is an empty list. If the reference MultiLayerRoutingArea is the top level MultiLayerRoutingArea then the response will only include a list of all of the top level Connections supporting the Call.
- CallID and sNPOrSNPPID: The target OS should return the routes for the Connection(s) that originate from the referenced SNP or SNPP of the referenced Call. If the SNP or SNPP is at the hierarchical level of a Routing Node, then the response is an empty list. If the SNP or SNPP is at the top level then the response will only include a list of all of the top level Connections, that originate from the specified SNP(P), supporting the Call.

In all cases the output is filtered appropriately by routeType.
All Routing Node Connections for a particular route will be returned regardless of any sharing of the same Routing Node Connections among distinct routes of the same Call.
The response may include the edge and internal MLSNPPLinks.

Note: given an SNCP scheme, the replied home route will always include both main and spare legs of the SNCP protection, while the replied actual route may return only one leg, if only one leg is currently active in the network. Once the CP routing succeeds to activate the spare leg, then both legs are replied when actual route is asked.

Additional information on exceptions:
unableToComply - Raised if the target OS can not access the route information.

Traceability: R_TMF518_MRI_II_xxx.

 getConnectionsAndRouteDetails?? 
Public  getConnectionsAndRouteDetails??(String, ConnectionAndRoute, , Boolean, RouteType, String )

This operation returns the routes of the Connections that support top level Connections of the specified Call.

The request may be tailored by the following allowed input combinations:

In the case of a 2 level routing area hierarchy:
- CallID only: the target OS should return the top level connections supporting the Call and their detailed routes.
- CallID and sNPOrSNPPID: The target OS should return the routes for the Connection(s) that originate from the referenced SNP or SNPP for the referenced Call. If the SNP or SNPP is at the hierarchical level of a Routing Node, then the response is an empty list.

In the case of a 3 level routing area hierarchy:
- CallID only: The target OS should return the routes for all connections that support any top level Connection of the specified Call for all MLRAs directly managed by the target OS.
- CallID and mLRAName: The target OS should return the routes for all connections within the referenced MultiLayerRoutingArea that supports any top level Connections of the referenced Call (there could be more than one top level Connection for the same Call passing through a MultiLayerRoutingArea). If the referenced MultiLayerRoutingArea is a Routing Node, then the response is an empty list. If the reference MultiLayerRoutingArea is the top level MultiLayerRoutingArea then the response will only include a list of all of the top level Connections supporting the Call.
- CallID and sNPOrSNPPID: The target OS should return the routes for the Connection(s) that originate from the referenced SNP or SNPP of the referenced Call. If the SNP or SNPP is at the hierarchical level of a Routing Node, then the response is an empty list. If the SNP or SNPP is at the top level then the response will only include a list of all of the top level Connections, that originate from the specified SNP(P), supporting the Call.

In all cases the output is filtered appropriately by routeType.
All Routing Node Connections for a particular route will be returned regardless of any sharing of the same Routing Node Connections among distinct routes of the same Call.
The response may include the edge and internal MLSNPPLinks.

Note: given an SNCP scheme, the replied home route will always include both main and spare legs of the SNCP protection, while the replied actual route may return only one leg, if only one leg is currently active in the network. Once the CP routing succeeds to activate the spare leg, then both legs are replied when actual route is asked.

Additional information on exceptions:
unableToComply - Raised if the target OS can not access the route information.

Traceability: R_TMF518_MRI_II_xxx.

 getRoute 
Public  getRoute(, Boolean,  )

This operation returns the route for the Connection. The route information, in the form of cross-connections, should identify the resources allocated to the Connection at the time of the request and the way in which these resources are used.
If the Connection has more alternative routes, then it retrieves the intended route if the Connection is partial, otherwise it retrieves the active route. This because there can be more partial routes and no active route, in such case the intended route is replied, as it is not known which of the partial routes is the candidate for becoming the active route.
The requesting OS specifies if it wants to retrieve only resources in the layer of the Connection (the cross-connections that belong to the Connection), or higher order cross-connections also (the cross-connections of other Connections that are used to carry the traffic of the queried connection).
All cross-connections allocated to the connection must be returned, even if they are not active in the managed elements.

This allows the requesting OS to query the route that will be used before activating an SNC.

Traceability: R_TMF518_MRI_II_0175, UC_TMF518_MRI_0027.
