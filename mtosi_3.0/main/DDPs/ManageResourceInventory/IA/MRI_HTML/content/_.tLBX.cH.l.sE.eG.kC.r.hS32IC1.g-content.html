Interface MultiLayerRoutingAreaRetrievalService

Operations
 getAllMultiLayerRoutingAreasWrtOs

This operation returns all Multi Layer Routing Areas (MLRAs) managed by the provided subordinate OS AND contained by the provided Management Domain (MD).
If no MD is provided all MLRAs managed by the provided subordinate OS are returned.
Note that "Wrt" in the name of the operation stands for "with respect to".

Traceability: R_TMF518_MRI_II_0181, UC_TMF518_MRI_0019.
 getAllMultiLayerRoutingAreas

This operation returns all Multi Layer Routing Areas (MLRAs) contained by the provided Management Domain (MD).
The returned list contains all the MLRAs at all levels of cotainment.

Traceability: R_TMF518_MRI_II_0181, UC_TMF518_MRI_0019.
 getAllTopLevelMultiLayerRoutingAreas

This operation returns all top level Multi Layer Routing Areas contained by a provided Management Domain.

Traceability: R_TMF518_MRI_II_0184, UC_TMF518_MRI_0019.
 getMultiLayerRoutingArea

This operation retrieves the details of a Multi Layer Routing Area given its name.

Traceability: R_TMF518_MRI_II_0182, UC_TMF518_MRI_0019.
 getAllSubordinateMultiLayerRoutingAreas

This operation provides a way for the requesting OS to return all the MLRAs that are "contained" within the targeted MultiLayerRoutingArea. This operation returns all the MLRAs down to and including those that represent Routing Nodes.
This allows a requesting OS to request the MLRAs or Routing Nodes that are completely managed by the target OS (i.e. the target OS is allowed to read and provision all transport plane aspects, e.g. PTP, CTP).
If the operation is addressed to an MLRA of Routing Node type, which is at the bottom of the routing hierarchy, the result is an empty list.

Traceability: R_TMF518_MRI_II_0183, UC_TMF518_MRI_0020.
 getAllSubordinateRoutingAreaIdsWithConnection

This operation returns the identifiers of the subordinate MultiLayerRoutingAreas, at the lower partitioning level (N-1 level) and which are known to be involved in the route of a Connection at level N.

This operation supports a parameter that enables the specification of the type of route for which the subordinate MultiLayerRoutingAreas are being requested. This operation result is not recursive i.e. the operation has been designed to operate in a 3 level hierarchy, i.e. where the top level MultiLayerRoutingArea contains subordinate MultiLayerRoutingAreas, and such subordinate MultiLayerRoutingAreas contain Routing Nodes.
This operation can only be addressed to a top level routing area.

The output of this operation has the following characteristics:
- The MultiLayerRoutingArea List is empty if there is no MultiLayerRoutingAreas for the requested intended or actual route or if the MultiLayerRoutingArea is a MultiLayerRoutingNode.
- There may be no Home/Intended route in the case where none was provisioned.
- There may be no Actual route where the connection is searching or the actual route traverses a different MultiLayerRoutingArea to the Home/Intended route.
- For an SNCP connection, the replied Home/Intended route will always include both main and spare legs of the SNCP protection, while the replied actual route may return only one leg, if only such leg is currently active in the network.
- The route is described in full across the MultiLayerRoutingArea regardless of sharing of same Routing Node Connections among distinct routes of the same Call.

Additional information on exceptions:
unableToComply - Raised if the target OS does not control the originating point for the specified connection. Raised if the adressed MLRA does not represent a top level MLRA. Raised if the addressed MLRA is the top level MLRA but the network has only 2 levels of routing hierarchy.
notInValidState - Raised if the target OS is unable to comply because the Connection State is Searching.

Traceability: R_TMF518_MRI_II_0172, UC_TMF518_MRI_0026.
