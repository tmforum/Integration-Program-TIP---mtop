Interface AlarmRetrievalService

Operations
 getActiveAlarmsCount

This operation asks the target top-level OS to count the number of active Alarms as restricted by the provided filter. The target OS sends the result to the requesting OS.
The operation can only be directed to a top-level OS and not to a subordinate OS.

Traceability: R_TMF518_RTM_II_0036, UC_TMF518_RTM_0009.
 getActiveAlarms

This operation returns a specified subset of the active Alarms known to the target OS. The target OS returns all alarms satisfying the filter constraints provided by the requesting OS.
This operation can only be directed to a top-level OS and not to a subordinate OS.

Traceability: R_TMF518_RTM_II_0037, UC_TMF518_RTM_0010.

 getActiveAlarms 
Public  getActiveAlarms(, ActiveAlarmFilter,  )

This operation returns a specified subset of the active Alarms known to the target OS. The target OS returns all alarms satisfying the filter constraints provided by the requesting OS.
This operation can only be directed to a top-level OS and not to a subordinate OS.

Traceability: R_TMF518_RTM_II_0037, UC_TMF518_RTM_0010.

Parameters:
alarmReply [*]This parameter contains the returned Alarm information. The information is returned in one or more batches according to the scenario described in the following Use Cases of the MTOSI Business Agreement:
- UC_MTOSI_22: Active Alarms Retrieval - Scenario 1
- UC_MTOSI_23: Active Alarms Retrieval - Scenario 2
- UC_MTOSI_24: Active Alarms Retrieval - Scenario 3.
ActiveAlarmFilter filter [1]This parameter defines the subset of the set of active Alarms known to the target OS that are to be returned to the requesting OS.
whereAndHowToReply [1]The parameter contains the necessary information (including where and how) the target OS needs to send the response. The information provided in this parameter depends on the scenario (i.e., type of communication style) used for retrieving the alarms. Refer to the attached SD2-5_Communication_Styles supporting document. The individual information to be provided by the requesting OS is described in the following Use Cases of the MTOSI Business Agreement:
- UC_MTOSI_22: Active Alarms Retrieval - Scenario 1
- UC_MTOSI_23: Active Alarms Retrieval - Scenario 2
- UC_MTOSI_24: Active Alarms Retrieval - Scenario 3.
Returns:
void [1]
