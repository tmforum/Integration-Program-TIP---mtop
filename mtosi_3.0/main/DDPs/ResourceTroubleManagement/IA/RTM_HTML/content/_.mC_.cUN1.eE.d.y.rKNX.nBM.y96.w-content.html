Interface ProtectionControlService

Operations
 performProtectionCommand

This operation is used to execute a protection switch. The protection switch may be performed via a protection switch command, on a protection group or on a CTP/FTP involved in an SNCP. It performs a specified protection command or clears all outstanding persistent protection commands.
Note: At a PGP, for 4-fiber SONET/SDH SPRINGS, the span switch is performed on the MSP_1_FOR_N protection group, and the ring switch on the 4_FIBER_BLSR group.

Traceability: R_TMF518_RTM_II_0058, UC_TMF518_RTM_0008.
 createProtectionGroup

This operation creates a Protection Group within the addressed Managed Element.
The name, i.e., a unique identifier of the new Protection Group can be provided by the requesting OS. If no name is provided by the requesting OS, the target OS has to define a unique name.

The target OS may or may not allow for multiple Protection Groups of the same type, rate and an overlapping set of TPs. When multiple Protection Groups are supported, an attempt to create a new Protection Group with a type, layer and list of TPs fully or partially matching an existing Protection Group should not trigger an Exception, and a new Protection Group object with a unique name (selected by the requesting OS or target OS) shall be created. When multiple Protection Groups are not supported by the target OS, an existing Protection Group is returned when a requested name, rate, type and the list of TPs match an existing object, otherwise an exception EXCPT_OBJECT_IN_USE is thrown.
Existing Protection Groups are never altered by this method.
The cardinality of the terminationPointList in the protection GroupCreateData should be greater than 0 and must not exceed the protection group type specific maximum number of TP members; e.g., with 1:1 type maximum of two TPs can be included in the terminationPointList.

The resulting Protection Group must include at least one TP otherwise this operation fails and an exception shall be thrown. Existing Protection Groups are never altered by this operation.

Additional information on exceptions:
invalidInput - Raised when any input parameter is syntactically incorrect (e.g. a field of protectionGroupCreateData is invalid including any field in the terminationPointList).
This exception is also raised when parameters are syntactically correct but terminationPointList is empty or the cardinality of terminationPointList in the protectionGroupCreateData exceeds the maximum number of TP members allowed for a specific a protection group type.
The exception is raised when TPs in the protectionGroupList do not support a layer specified in the protectionGroupCreateData or cannot be associated in a protection relation at this layer.

unableToComply - Raised when the target OS is unable to process the request because at least one parameter in the protectionGroupCreateData although valid and syntactically correct could not be set, or when the request fails at the Managed Eelement.

Traceability: R_TMF518_RTM_II_0070, R_TMF518_RTM_II_0071, UC_TMF518_RTM_0028.
 deleteProtectionGroup

This operation deletes a Protection Group.

Additional information on exceptions:
unableToComply - This exception is raised and delete operation is rejected when swap of protection role is selected (case 1) but protectionGroupType does not support the feature, or client TPs protected by the Protection Group being deleted do not support relaxed containment feature, hence these client TPs should change their object names.

Traceability: R_TMF518_RTM_II_0074, UC_TMF518_RTM_0029.
 modifyProtectionGroup

This operation modifies an existing Protection Group.
The terminationPointList should always include the complete intended list of Protection Group TP names rather than a delta from the existing Protection Group as each TP has well defined position in the list according to their role and priority. The terminationPointList should never be empty as Protection Groups without TP members are not permitted. The list size should not exceed the cardinality specific to the protectionGroupType with an exception of a reconfiguration scenario when working or protecting TP may be replaced thus existing followed by replacement TP is included in the list.
The swap of protection role can be provisioned by modifying accordingly the positions of TPs in the list. Note that this feature is applicable to 1+1 or 1:N protection group type only.
When all attributes including terminationPointList are exactly the same as in the target Protection Group, there should be no disruption to traffic and the modify operation should complete successfully. This operation should be utilized along with protection commands to replace and/or swap workers with protecting TPs. The consequences of using this operation only without steps to execute Lockout, protection switch and lock release are implementation/target OS dependent.


Additional information on exceptions:
invalidInput - Raised when any input parameter is syntactically incorrect (e.g., a field of protectionGroupCreateData is invalid including any field in the terminationPointList).
The exception is also raised under the following conditions:
- TPs in the terminationPointList do not support the same layer as the Protection Group or cannot be associated in a protection relation at this layer
- the terminationPointList contains no TPs.

unableToComply - Raised when the target OS is unable to process the request because at least one parameter in the protectionGroupModifyData although valid and syntactically correct could not be set, or when the request fails at the Managed Eelement.
The exception is raised also when swap of protection role is provisioned (through change of TP positions in the terminationPointList ) but protectionGroupType does not support the feature, or client TPs protected by the Protection Group do not support the relaxed containment feature, hence these client TPs should change their object names.

Traceability: R_TMF518_RTM_II_0072, R_TMF518_RTM_II_0073, UC_TMF518_RTM_0030.
